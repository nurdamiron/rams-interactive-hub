## 🔴 Критичные проблемы (из-за них «лента не работает»)

### 1. Неверный маппинг режимов в Electron
**Файл:** `electron/main.js:1166-1169`

```js
const modeToFwId = {
  'WAVE': 3, 'PULSE': 4, 'SPARKLE': 5, 'FIRE': 6,
  'CHASE': 3, 'STATIC': 3, 'RAINBOW': 3, 'METEOR': 3,
};
```

Этот маппинг написан под **старую прошивку v3.2** (см. комментарий в `lib/esp32-client.ts:125`). На esp32_master: `3=Chase`, `4=Sparkle`, `5=Wave`. То есть просишь «Волну» — получаешь «Бег», просишь «Пульс» — получаешь «Искры», а STATIC/RAINBOW/METEOR все дают Chase.

**Фикс:**
```js
const modeToFwId = {
  'STATIC': 0, 'PULSE': 1, 'RAINBOW': 2, 'CHASE': 3,
  'SPARKLE': 4, 'WAVE': 5, 'FIRE': 6, 'METEOR': 7,
};
```

### 2. Режим RAINBOW ломает сам себя (главная причина «мёртвой» ленты в простое)
**Файлы:** `electron/main.js:1180-1188, 1227-1234` + `components/screensaver/screensaver.tsx:107`

Что происходит при включении скринсейвера (простой ≈ всё время работы стенда):
1. Скринсейвер вызывает `setLedMode("RAINBOW")`.
2. Electron ставит эффект `id=3` (на новой прошивке это **Chase**, а не Wave).
3. Запускает таймер, который **каждые 150мс** шлёт `/api/color`.
4. Обработчик `/api/color` в прошивке **принудительно переключает режим в STATIC** (`main.cpp:314`).

Итог: вместо «волна + радуга» лента показывает **статичный цвет** с медленной сменой оттенка, а ESP32 WebServer получает ~7 запросов/сек, что душит остальные команды (блоки, эффекты). Визуально лента выглядит замершей/сломанной.

**Фикс:** удалить `startRainbowCycle`/`stopRainbowCycle` полностью. У прошивки есть родной эффект Rainbow (`id=2`) с плавной анимацией цвета — RAINBOW-режим должен просто слать `/api/effect?id=2`.

### 3. Рассинхрон состояния авто-цикла между Electron и прошивкой
**Файл:** `electron/main.js:1163, 1201-1208, 1215`

Прошивка стартует с `autoCycleEnabled=true`, а в Electron флаг `firmwareAutoCycleOn=false`. Последствия:

- **Выбор ручного эффекта не отключает авто-цикл**: `stopLedAutoCycle()` шлёт `enabled=0` только если флаг true. Флаг false → команда не уходит → через ≤60с прошивка сама перещёлкивает эффект. Симптом: «выбрал эффект, а через минуту он сам сменился».
- **Первый клик AUTO в админке не выключает, а включает уже включённое** (тумблер работает наоборот, пока состояние не совпадёт случайно).
- После рестарта Electron рассинхрон возникает снова.

**Фикс:** при старте Electron и перед каждым переключением синхронизироваться: `GET /api/autocycle` → выставить `firmwareAutoCycleOn` из ответа. Либо проще: в `stopLedAutoCycle()` слать `enabled=0` **безусловно**.

### 4. Клиент не понимает формат `/api/status` новой прошивки
**Файлы:** `lib/esp32-client.ts:16-21, 68-78, 346` vs `main.cpp:143-169`

Клиент ждёт `{active: number, blocks: number[]}`. Прошивка отдаёт `{activeBlocks: number, blocks: ["stop","up",...], mega1: "ok"}`.

- `isBlockActive()` в `components/actuator-control.tsx:102` делает `blocks.includes(blockNum)` по массиву **строк** → всегда false → подсветка активных блоков в панели не работает.
- `discoverESP32()` (`lib/esp32-client.ts:346`) проверяет `data.hasOwnProperty("active")` → новая прошивка **не проходит проверку** → автопоиск ESP32 в браузерном режиме никогда его не найдёт.

**Фикс:** обновить `ESP32Status` и парсинг: `activeBlocks`, `blocks` как массив строк (или конвертировать в индексы поднятых), discovery проверять по `activeBlocks`/`apIP`.

### 5. Захардкоженный IP в Next.js-прокси (браузерный режим)
**Файл:** `next.config.ts:17`

```ts
destination: 'http://192.168.110.65/api/:path*',
```

ESP32 получает адрес от роутера по DHCP — после перезагрузки роутера IP может смениться, и **весь браузерный режим теряет железо** (Electron-режим спасает авто-discovery, у веба его нет из-за п.4).

**Фикс (по нарастанию надёжности):**
1. Прописать статический DHCP-lease для MAC ESP32 на роутере.
2. Использовать mDNS: `http://RAMS-ESP32.local/api/:path*` (hostname уже задан в OTA-настройках прошивки).
3. Заменить rewrite на API-роут `/app/api/esp32/[...path]/route.ts` с IP из конфига/env — тогда адрес меняется без пересборки.

## 🟡 Средние проблемы

### 6. Третий, тоже неверный маппинг в legacy-командах
**Файл:** `electron/main.js:1313`

```js
{ 'STATIC': 2, 'PULSE': 1, 'RAINBOW': 0, 'CHASE': 3, 'SPARKLE': 5, 'WAVE': 3, 'FIRE': 6, 'METEOR': 7, 'OFF': 4 }
```
Здесь `OFF: 4` = **Sparkle**, `STATIC: 2` = Rainbow. Это третий независимый маппинг в кодовой базе. Удалить legacy-путь `hardware-send-command LED:MODE:*` или привести к единому enum (лучше: один экспортируемый словарь).

### 7. Три конкурирующих авто-цикла
- Прошивка: родной автоцикл (вкл. по умолчанию).
- `components/actuator-control.tsx:179-206`: свой `setInterval`.
- `components/controls/effect-control.tsx:50-71`: ещё один свой `setInterval`.

Клиентские циклы шлют `/api/effect` поверх фирменного → эффекты прыгают непредсказуемо. **Фикс:** оба UI-тумблера должны дергать `/api/autocycle` (см. п.8) и отображать реальное состояние из `GET /api/autocycle`, клиентские таймеры удалить.

### 8. В `ESP32Client` нет методов для `/api/autocycle` и `/api/state`
**Файл:** `lib/esp32-client.ts`

Браузерный режим вообще не может управлять авто-циклом и не может узнать текущее состояние ленты. Добавить:

```ts
async setAutoCycle(enabled: boolean, intervalSec = 60): Promise<void>
async getAutoCycle(): Promise<{autoCycle: boolean; interval: number; currentEffect: number}>
async getState(): Promise<{r,g,b,bri,spd,fx,autoCycle,autoCycleInterval}>
```

### 9. Логика скринсейвера
**Файл:** `components/screensaver/screensaver.tsx:104-111`

Вход → RAINBOW (сломан, п.2), выход → WAVE (превращается в Chase, п.1). Рекомендуемая политика после фиксов:
- Вход в скринсейвер → `/api/autocycle?enabled=1` (пусть прошивка сама крутит все 8 эффектов).
- Выход → `enabled=0` + эффект WAVE (`id=5`).

## Быстрая диагностика (прямо сейчас, 1 минута)

```bash
# Состояние ленты: fx = текущий эффект, bri = яркость, autoCycle = true/false
curl http://192.168.110.65/api/state

# Если bri: 0 — кто-то нажал OFF; вернуть:
curl -X POST "http://192.168.110.65/api/bri?v=200"

# Принудительно включить родной автоцикл:
curl -X POST "http://192.168.110.65/api/autocycle?enabled=1&interval=60"
```

Если `/api/state` не отвечает по этому IP — ESP32 сменил адрес: подключись к AP `RAMS-ESP32` (пароль `rams2024`) и проверь `http://192.168.4.1/api/status`, поле `staIP` покажет актуальный адрес в сети роутера.
